## 1. บทนำและบริบททางธุรกิจ (Business Context)

### 1.1 ปัญหาที่ต้องการแก้ไข (Problem Statement)

ปัจจุบันสถานีตำรวจทั่วประเทศไทย **1,500 แห่ง** ยังไม่มีระบบจัดคิวที่เป็นมาตรฐาน ทำให้เกิดปัญหาซ้ำๆ ดังนี้:

1. **ประชาชนไม่ทราบลำดับ** — เดินเข้ามาแล้วไม่รู้ว่าต้องรอกี่คน ถึงคิวเมื่อไหร่ ทำให้รู้สึกไม่มั่นใจและเสียเวลา
2. **ไม่มีการแยกลำดับความสำคัญ** — ผู้สูงอายุ ผู้พิการ หรือผู้ที่มีเหตุฉุกเฉิน ต้องรอเหมือนกรณีทั่วไป
3. **ไม่มีข้อมูลสถิติ** — ผู้บริหารสถานีไม่มีตัวเลขจำนวนผู้มาใช้บริการ เวลารอเฉลี่ย ชั่วโมงที่คนเยอะ ทำให้จัดสรรกำลังพลไม่ตรงจุด
4. **วุ่นวายและไม่เป็นระเบียบ** — ประชาชนต้องถามเจ้าหน้าที่ซ้ำๆ ว่า "ต้องไปไหน" "ถึงคิวหรือยัง" เป็นภาระกับทั้งสองฝ่าย
5. **ข้อมูลสูญหายเมื่อระบบมีปัญหา** — ถ้าใช้กระดาษหรือระบบออนไลน์อย่างเดียว เมื่อเน็ตหลุดหรือไฟดับ ข้อมูลหายทั้งหมด

### 1.2 วัตถุประสงค์ทางธุรกิจ (Business Objectives)

| #    | วัตถุประสงค์                                              | ตัวชี้วัดความสำเร็จ (KPI)                                 |
| ---- | --------------------------------------------------------- | --------------------------------------------------------- |
| BO-1 | ลดเวลารอเฉลี่ยของประชาชน                                  | เวลารอเฉลี่ย ≤ 20 นาที/คิว (ยกเว้นหมวดพบพนักงานสอบสวน)    |
| BO-2 | ประชาชนสามารถกดรับคิวเองได้โดยไม่ต้องมีเจ้าหน้าที่คอยช่วย | อัตราการใช้งานด้วยตนเอง (Self-Service Rate) ≥ 80%         |
| BO-3 | มีข้อมูลสถิติเพื่อจัดสรรกำลังพลและทรัพยากร                | ผู้บริหารสถานีเข้าถึงรายงานได้ทุกวัน                      |
| BO-4 | ส่วนกลางเห็นภาพรวมคิวทั้งประเทศ (1,500 สถานี)             | Dashboard ส่วนกลางแสดงข้อมูลรวมได้ (รายวัน/สัปดาห์/เดือน) |
| BO-5 | ระบบต้องทำงานต่อเนื่องไม่หยุดชะงัก แม้เครือข่ายขัดข้อง    | Downtime = 0 ในส่วนการออกคิว (Offline-First Architecture) |
| BO-6 | ดูแลผู้สูงอายุ/ผู้ที่ต้องได้รับสิทธิพิเศษอย่างเป็นระบบ    | คิว VIP ถูกเรียกก่อน Normal ภายในประเภทเดียวกันเสมอ       |

## 3. ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

### 3.1 ประเภทบริการ (Service Types)

| รหัส   | ประเภทบริการ          | Prefix | ตัวอย่างเลขคิว | คำอธิบาย                                        |
| ------ | --------------------- | ------ | -------------- | ----------------------------------------------- |
| SVC-01 | แจ้งความร้องทุกข์     | `R`    | R-001, R-042   | ร้องทุกข์ แจ้งเอกสารหาย แจ้งเหตุทั่วไป          |
| SVC-02 | พบพนักงานสอบสวน       | `S`    | S-001, S-015   | นัดพบสอบปากคำ ให้การเพิ่มเติม ติดตามคดี         |
| SVC-03 | เหตุด่วน / อุบัติเหตุ | `E`    | E-001, E-003   | แจ้งเหตุฉุกเฉิน อุบัติเหตุ เหตุร้ายที่เพิ่งเกิด |
| SVC-04 | ปรึกษากฎหมาย          | `L`    | L-001, L-020   | สอบถามข้อกฎหมาย ขอคำปรึกษาเบื้องต้น             |

### 3.2 ระดับความสำคัญ (Priority Levels)

Priority แยก **ภายในแต่ละประเภทบริการ** (เช่น VIP ของ SVC-01 กับ VIP ของ SVC-03 เป็นคนละคิว)

| ระดับ | Array Index   | ชื่อ                 | เงื่อนไข                                              | ตัวอย่าง |
| ----- | ------------- | -------------------- | ----------------------------------------------------- | -------- |
| 0     | `priority[0]` | **VIP** (สิทธิพิเศษ) | ผู้สูงอายุ (≥ 60 ปี), ผู้พิการ, สตรีมีครรภ์, พระภิกษุ | R-V001   |
| 1     | `priority[1]` | **Normal** (ปกติ)    | ประชาชนทั่วไป                                         | R-001    |

> **กฎ:** ภายในประเภทบริการเดียวกัน VIP จะถูกเรียกก่อน Normal เสมอ (Strict Priority)
> ภายในระดับเดียวกัน ใช้ **FIFO** (มาก่อนได้ก่อน)

### 3.3 โครงสร้างการจัดคิว (Queue Matrix)

ระบบจะมีคิวทั้งหมด **8 คิว** (4 ประเภทบริการ × 2 ระดับ Priority):

```
             SVC-01 (แจ้งความ)   SVC-02 (สอบสวน)   SVC-03 (เหตุด่วน)   SVC-04 (กฎหมาย)
            ┌─────────────────┬─────────────────┬──────────────────┬─────────────────┐
priority[0] │  VIP แจ้งความ   │  VIP สอบสวน     │  VIP เหตุด่วน    │  VIP กฎหมาย     │
(VIP)       │  R-V001→R-V002  │  S-V001→S-V002  │  E-V001→E-V002   │  L-V001→L-V002  │
            ├─────────────────┼─────────────────┼──────────────────┼─────────────────┤
priority[1] │  Normal แจ้งความ│  Normal สอบสวน  │  Normal เหตุด่วน │  Normal กฎหมาย  │
(Normal)    │  R-001→R-002    │  S-001→S-002    │  E-001→E-002     │  L-001→L-002    │
            └─────────────────┴─────────────────┴──────────────────┴─────────────────┘
```

> **ผลกระทบต่อ Architecture:** โครงสร้าง Array of Linked Lists ต้องขยายเป็น **2 มิติ** — `queues[service_type][priority_level]` แทนที่จะเป็น `queues[priority_level]` แบบเดิม

### 3.4 Use Cases (กรณีการใช้งาน)

#### UC-01: ประชาชนกดรับคิวเอง (Self-Service Enqueue)

| หัวข้อ                | รายละเอียด                                                                                           |
| --------------------- | ---------------------------------------------------------------------------------------------------- |
| **ผู้ใช้**            | U-01 ประชาชน (หรือ U-03 เจ้าหน้าที่ช่วยกด)                                                           |
| **จุดใช้งาน**         | ตู้ Kiosk / Tablet หน้าสถานีตำรวจ                                                                    |
| **เงื่อนไขเบื้องต้น** | ตู้ Kiosk เปิดทำงานอยู่                                                                              |
| **ขั้นตอนหลัก**       | 1. ประชาชนเดินมาที่ตู้ Kiosk                                                                         |
|                       | 2. หน้าจอแสดง 4 ปุ่มบริการ: แจ้งความร้องทุกข์ / พบพนักงานสอบสวน / เหตุด่วน-อุบัติเหตุ / ปรึกษากฎหมาย |
|                       | 3. ประชาชนกดเลือกประเภทบริการ                                                                        |
|                       | 4. ระบบถามว่าเป็นผู้สูงอายุ/ผู้พิการ/สตรีมีครรภ์หรือไม่ (ปุ่ม "ใช่" / "ไม่")                         |
|                       | 5. ระบบสร้างเลขคิว (เช่น R-V003 หรือ R-021)                                                          |
|                       | 6. ระบบพิมพ์สลิปคิว (มีเลขคิว + ประเภทบริการ + จำนวนคิวข้างหน้า + QR code)                           |
|                       | 7. ข้อมูลถูก Sync ขึ้น Server (Background)                                                           |
| **ผลลัพธ์**           | ประชาชนได้สลิปคิว, รู้จำนวนคิวข้างหน้า                                                               |
| **กรณีพิเศษ**         | เครือข่ายขัดข้อง → บันทึก Local DB + พิมพ์สลิปได้ปกติ (Offline Mode)                                 |
| **ข้อกำหนด UI**       | ปุ่มต้องใหญ่ ตัวอักษรอ่านง่าย เหมาะกับผู้สูงอายุ ไม่ต้อง Login ไม่ต้องกรอกข้อมูลส่วนตัว              |

#### UC-02: เจ้าหน้าที่เรียกคิวถัดไป (Call Next Queue)

| หัวข้อ                | รายละเอียด                                                                       |
| --------------------- | -------------------------------------------------------------------------------- |
| **ผู้ใช้**            | U-02 เจ้าหน้าที่ประจำช่องบริการ                                                  |
| **เงื่อนไขเบื้องต้น** | เจ้าหน้าที่ Login เข้าระบบ + ช่องบริการว่าง                                      |
| **ขั้นตอนหลัก**       | 1. เจ้าหน้าที่กดปุ่ม "เรียกคิวถัดไป"                                             |
|                       | 2. ระบบดึงคิวตามลำดับ: VIP ก่อน → Normal (ภายในประเภทบริการที่ช่องนั้นรับผิดชอบ) |
|                       | 3. หน้าจอ Display Board อัปเดตทันที + เสียงเรียกดัง                              |
|                       | 4. แสดง: "หมายเลข R-V003 เชิญที่ช่อง 2"                                          |
|                       | 5. ประชาชนเดินเข้ามา → เจ้าหน้าที่กด "เริ่มให้บริการ"                            |
| **ผลลัพธ์**           | สถานะคิว: `pending` → `processing`                                               |
| **เสียงเรียก**        | เสียงพูดภาษาไทย: "หมายเลข [เลขคิว] เชิญที่ช่องบริการ [หมายเลข]"                  |

#### UC-03: จบบริการ (Complete Service)

| หัวข้อ          | รายละเอียด                                                                                               |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| **ผู้ใช้**      | U-02 เจ้าหน้าที่ประจำช่องบริการ                                                                          |
| **ขั้นตอนหลัก** | 1. เจ้าหน้าที่ให้บริการเสร็จ                                                                             |
|                 | 2. กดปุ่ม "เสร็จสิ้น"                                                                                    |
|                 | 3. สถานะคิว: `processing` → `completed`                                                                  |
|                 | 4. ระบบบันทึก: เวลารอ (wait_time), เวลาให้บริการ (service_duration), เจ้าหน้าที่ผู้ให้บริการ, ช่องบริการ |
|                 | 5. ช่องบริการพร้อมรับคิวถัดไป                                                                            |
| **ผลลัพธ์**     | คิวถูกปิด, ข้อมูลเข้าสู่ระบบรายงาน                                                                       |

#### UC-04: ข้ามคิว / พักคิว (Skip / Hold)

| หัวข้อ          | รายละเอียด                                                                       |
| --------------- | -------------------------------------------------------------------------------- |
| **ผู้ใช้**      | U-02 เจ้าหน้าที่ประจำช่องบริการ                                                  |
| **ขั้นตอนหลัก** | 1. เจ้าหน้าที่กด "เรียกคิวถัดไป" แล้วประชาชนไม่มา                                |
|                 | 2. ระบบรอ **3 นาที** แล้วเรียกซ้ำอัตโนมัติ (ครั้งที่ 2)                          |
|                 | 3. ถ้ายังไม่มา → รอ **3 นาที** แล้วเรียกซ้ำอัตโนมัติ (ครั้งที่ 3 — ครั้งสุดท้าย) |
|                 | 4. ถ้ายังไม่มาอีก → สถานะเปลี่ยนเป็น `skipped` อัตโนมัติ                         |
|                 | 5. เจ้าหน้าที่สามารถกด "ข้ามทันที" ได้ โดยไม่ต้องรอ                              |
| **กฎธุรกิจ**    | เรียกซ้ำสูงสุด **3 ครั้ง** (รวมครั้งแรก) ห่างกัน **3 นาที**                      |
|                 | คิวที่ถูกข้ามจะ **ไม่กลับเข้าคิวอัตโนมัติ** — ประชาชนต้องมากดคิวใหม่             |

#### UC-05: ยกเลิกคิว (Cancel Queue)

| หัวข้อ                            | รายละเอียด                                                      |
| --------------------------------- | --------------------------------------------------------------- |
| **ผู้ใช้**                        | U-02 เจ้าหน้าที่ หรือ U-01 ประชาชน (จากตู้ Kiosk)               |
| **ขั้นตอนหลัก — ฝั่งประชาชน**     | 1. ประชาชนสแกน QR Code บนสลิป ที่ตู้ Kiosk                      |
|                                   | 2. ระบบแสดงข้อมูลคิวของตน                                       |
|                                   | 3. กดปุ่ม "ยกเลิกคิว"                                           |
|                                   | 4. ระบบยืนยัน → สถานะ: `cancelled`                              |
| **ขั้นตอนหลัก — ฝั่งเจ้าหน้าที่** | 1. เจ้าหน้าที่ค้นหาเลขคิว                                       |
|                                   | 2. กดปุ่ม "ยกเลิก"                                              |

## 6. สถานะคิว (Queue State Machine)

```
                    ┌──────────────────────────────────────────────────┐
                    │                                         (โอนคิว) │
                    ▼                                                  │
  [กดรับคิว] ──► pending ──► processing ──► completed                  │
                  │ │ ▲            │                                    │
                  │ │ │            ├──► transferred ────────────────────┘
                  │ │ │            │     (สร้างคิวใหม่เป็น VIP ปลายทาง)
                  │ │ └────────────┘
                  │ │   (เรียกซ้ำ)
                  │ ▼
                  │ on_hold ───► skipped (เรียกครบ 3 ครั้ง)
                  │ │
                  ▼ ▼
               cancelled
          (จาก pending / on_hold เท่านั้น)
```

**ตารางสถานะ:**

| สถานะ          | ความหมาย                       | เปลี่ยนไปเป็นได้                  | Trigger                                    |
| -------------- | ------------------------------ | --------------------------------- | ------------------------------------------ |
| `pending`      | รอเรียก                        | processing, on_hold, cancelled    | —                                          |
| `processing`   | กำลังให้บริการ                 | completed, transferred            | เจ้าหน้าที่กด "เริ่มบริการ"                |
| `completed`    | เสร็จสิ้น                      | — (Final)                         | เจ้าหน้าที่กด "เสร็จสิ้น"                  |
| `on_hold`      | พักคิว (เรียกแล้วไม่มา)        | pending (เรียกซ้ำ), skipped, cancelled | ระบบเรียกแล้วไม่มีคนมา                |
| `skipped`      | ข้ามแล้ว (เรียกครบ 3 ครั้ง)    | — (Final)                         | เรียกครบ 3 ครั้ง หรือ เจ้าหน้าที่กด "ข้าม" |
| `transferred`  | โอนไปบริการอื่น                | — (Final, มีคิวใหม่แทน)           | เจ้าหน้าที่กด "โอน"                        |
| `cancelled`    | ยกเลิก                         | — (Final)                         | ประชาชน/เจ้าหน้าที่ กด "ยกเลิก"            |
| `pending_sync` | (Internal) รอ Sync ขึ้น Server | synced (อัตโนมัติ)                | เครือข่ายขัดข้อง                           |

## 7. ข้อมูลที่จัดเก็บ (Data Model Overview)

### 7.1 Queue Record (1 รายการต่อ 1 คิว)

| Field              | Type      | คำอธิบาย                                                                 |
| ------------------ | --------- | ------------------------------------------------------------------------ |
| `queue_id`         | UUID (PK) | Idempotency Key — ใช้ป้องกันคิวซ้ำ                                       |
| `queue_number`     | String    | เลขคิวที่แสดง เช่น R-V003                                                |
| `service_type`     | Enum      | SVC-01, SVC-02, SVC-03, SVC-04                                           |
| `priority_level`   | Integer   | 0 = VIP, 1 = Normal                                                      |
| `status`           | Enum      | pending, processing, completed, on_hold, skipped, transferred, cancelled |
| `created_at`       | Timestamp | เวลาที่กดรับคิว                                                          |
| `called_at`        | Timestamp | เวลาที่ถูกเรียกครั้งแรก (nullable)                                       |
| `started_at`       | Timestamp | เวลาที่เจ้าหน้าที่เริ่มให้บริการ (nullable)                              |
| `completed_at`     | Timestamp | เวลาที่จบบริการ (nullable)                                               |
| `counter_id`       | String    | ช่องบริการ (nullable)                                                    |
| `officer_id`       | String    | เจ้าหน้าที่ผู้ให้บริการ (nullable)                                       |
| `call_count`       | Integer   | จำนวนครั้งที่ถูกเรียก (Default: 0)                                       |
| `transferred_from` | UUID (FK) | อ้างอิงคิวต้นทางกรณีโอนมา (nullable)                                     |
| `station_id`       | String    | รหัสสถานีตำรวจ                                                           |
| `sync_status`      | Enum      | pending_sync, synced                                                     |

### 7.2 ตัวชี้วัดที่คำนวณได้ (Derived Metrics)

| ตัวชี้วัด                            | สูตร                        |
| ------------------------------------ | --------------------------- |
| **Wait Time** (เวลารอ)               | `called_at - created_at`    |
| **Service Duration** (เวลาให้บริการ) | `completed_at - started_at` |
| **Total Time** (เวลาทั้งหมด)         | `completed_at - created_at` |

## 10. สิ่งที่ Architecture ต้องออกแบบเพิ่ม (Gap Analysis)

จาก Traceability Matrix ด้านบน สรุปสิ่งที่ต้องเพิ่มใน Architecture Document:

| #      | Gap                           | รายละเอียด                                                                                       | ความสำคัญ |
| ------ | ----------------------------- | ------------------------------------------------------------------------------------------------ | --------- |
| GAP-01 | **Queue Matrix 2 มิติ**       | ขยาย Data Structure จาก `queues[priority]` เป็น `queues[service_type][priority]` — 4×2 = 8 คิว   | 🔴 สูง    |
| GAP-02 | **Kiosk Mode UI**             | ออกแบบ UI สำหรับตู้ Kiosk — ปุ่มใหญ่, ไม่ต้อง Login, ≤ 3 ขั้นตอน, Lock Kiosk                     | 🔴 สูง    |
| GAP-03 | **WebSocket / SSE Layer**     | สำหรับ Display Board Real-time + เสียงเรียกภาษาไทย (TTS / Pre-recorded)                          | 🔴 สูง    |
| GAP-04 | **State Machine (Skip/Hold)** | เพิ่มสถานะ on_hold, skipped + Auto-recall Timer (3 ครั้ง × 3 นาที)                               | 🟡 กลาง   |
| GAP-05 | **Transfer Logic**            | โอนคิวข้ามประเภท + Auto-VIP + Reference tracking                                                 | 🟡 กลาง   |
| GAP-06 | **Cancel Endpoint**           | ยกเลิกคิวผ่าน QR Scan + API                                                                      | 🟡 กลาง   |
| GAP-07 | **Reporting Module**          | Dashboard + รายงาน 6 ประเภท + Export CSV                                                         | 🟡 กลาง   |
| GAP-08 | **Auth & Role-based Access**  | Login สำหรับ Officer / Manager / Admin / HQ Executive (ประชาชนไม่ต้อง Login)                     | 🟡 กลาง   |
| GAP-09 | **Counter ↔ Service Mapping** | 1 ช่องรับได้หลายประเภท + Admin จัดการได้ + Validation ขั้นต่ำ 4 ช่อง (ประเภทละ 1)                | 🟢 ต่ำ    |
| GAP-10 | **Daily Auto-reset**          | Cron job รีเซ็ตเลขคิวทุกวัน 00:00                                                                | 🟢 ต่ำ    |
| GAP-11 | **Central Server**            | Server ส่วนกลางรับ Aggregate data จาก 1,500 สถานี (ทุก 15 นาที) เก็บสรุป รายวัน/สัปดาห์/เดือน    | 🔴 สูง    |
| GAP-12 | **HQ Dashboard**              | Web Dashboard สำหรับส่วนกลาง — Filter ตามจังหวัด/กองบังคับการ/สถานี, Export CSV/PDF              | 🔴 สูง    |
| GAP-13 | **Data Aggregation Pipeline** | Pipeline รวมข้อมูลจากทุกสถานี → คำนวณสรุป (จำนวนคิว, เวลารอเฉลี่ย, Peak Hours) → เก็บ Central DB | 🟡 กลาง   |
